# Contributing

このファイルは、R4B-beginners-luck 配下のリポジトリで共通して利用する開発ルールをまとめたものです。

## 1. 基本方針
- 小さく変更する
- 1つの変更に1つの目的を持たせる
- 無関係な変更を混ぜない
- レビューしやすい形で履歴を残す
- 困ったら厳密さよりも分かりやすさを優先する

## 2. ブランチ運用
このプロジェクトでは、以下のブランチを使用します。

| ブランチ名 | 役割 | 派生元 | マージ先 |
|---|---|---|---|
| `main` | 安定版・完成版を置くブランチ | - | - |
| `develop` | 開発中の内容をまとめるブランチ | `main` | `main` |
| `feature/*` | 機能ごとの開発を行うブランチ | `develop` | `develop` |

### 運用ルール
- `main` には直接 push しない
- 作業は基本的に `feature/*` ブランチで行う
- 作業完了後は `develop` にマージする
- ある程度まとまったら `develop` を `main` にマージする
- 1ブランチ1目的を原則とする

### feature ブランチ名の例
- `feature/top-page-ui`
- `feature/goal-map`
- `feature/calendar-view`
- `feature/login-form`

## 3. コミットルール
コミットメッセージは以下の形式を基本とします。

`type: 内容`

または

`type(scope): 内容`

### 使用する type
- `add`：新機能追加
- `fix`：バグ修正
- `update`：既存機能の改善
- `change`：仕様変更
- `clean`：整理・リファクタリング
- `remove`：削除
- `docs`：ドキュメント更新

### scope について
- scope は変更対象を分かりやすくするために付与する
- scope は必須ではない
- 付けられる場合は付ける
- 使用する scope は今後必要に応じて追加してよい
- 新しい scope を使う場合は、意味が分かる短い名前にする
- 表記ゆれを避けるため、既存の scope があればそれを優先する

### 使用する scope の例

#### 画面
- `top`：トップ画面
- `calendar`：カレンダー画面
- `goals`：目標一覧・目標マップ画面
- `login`：ログイン画面

#### 共通UI
- `layout`：レイアウト全体
- `modal`：モーダル関連
- `sidebar`：サイドバー関連
- `graph`：目標マップ・グラフ表示関連

#### データ・状態
- `types`：型定義
- `dummy`：ダミーデータ
- `state`：状態管理

#### バックエンド
- `api`：API関連
- `auth`：認証関連
- `db`：データベース関連

#### ドキュメント・設定
- `docs`：ドキュメント
- `org`：Organization共通ルール
- `config`：設定ファイル関連

### コミット例(scopeなし)
- `add: トップ画面の空状態カードを追加`
- `fix: カレンダーの横線表示を削除`
- `update: 目標マップの見た目を調整`
- `change: 短期目標の表示仕様を変更`
- `clean: GoalDetailPanel の処理を整理`
- `remove: 不要なダミーデータを削除`
- `docs: CONTRIBUTING.md を更新`

### コミット例(scopeあり)
- `add(top): 空状態カードを追加`
- `fix(calendar): 横線表示を削除`
- `update(goals): 目標マップの見た目を調整`
- `change(auth): ログイン画面の入力仕様を変更`
- `clean(graph): GoalDetailPanel の処理を整理`
- `remove(dummy): 不要なダミーデータを削除`
- `docs(org): PR ルールを追記`


### コミット時のルール
- 1コミット1内容を意識する
- 無関係な変更を1つのコミットに含めない
- 後から見て内容が分かるメッセージにする
- 大きすぎる変更は分けてコミットする

## 4. プッシュルール
- 区切りの良い単位で push する
- 明らかに壊れている状態では共有ブランチに push しない
- Pull Request 作成前に不要なファイルや差分が含まれていないか確認する

## 5. Pull Request ルール
Pull Request では、以下の内容を記載する。

- Purpose
- Changes
- Impact
- Checks

## 6. Pull Request 前チェック
- [ ] ビルドまたは動作確認を行った
- [ ] 不要な差分が含まれていない
- [ ] 変更内容を自分で説明できる
- [ ] AI生成コードを未確認のまま含めていない
- [ ] レビューしやすい粒度になっている
